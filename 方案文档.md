# 微信公众号文章自动生成工具 - 技术方案

## 1. 项目概述

### 1.1 目标

针对指定 URL 进行爬虫抓取，将内容送入 LLM（豆包大模型）进行分析优化，最终输出为微信公众号编辑器可直接粘贴使用的 HTML 文件。

### 1.2 核心流程

```
用户输入URL（支持单URL / 批量URL列表文件）
    ↓
爬虫抓取（trafilatura 为主，多级回退，反爬策略）
    ↓
LLM 分析优化（httpx 直调 ARK API，限量图文理解）
    ↓
自定义 MD 渲染器 → 微信兼容 HTML（内联样式，一步到位）
    ↓
保存至 result/{日期}/{文章标题}.html + .md
```

## 3. 技术选型

| 模块       | 技术                        | 说明                                                |
| ---------- | --------------------------- | --------------------------------------------------- |
| HTTP 请求  | `httpx`                     | 爬虫 + LLM API 调用统一使用，无需额外 SDK           |
| 正文提取   | `trafilatura`               | 主力提取器，准确率远高于手写规则                     |
| 备选提取   | `readability-lxml` / `newspaper4k` | trafilatura 失败时的第二/第三备选            |
| HTML 辅助  | `BeautifulSoup4`            | 图片/视频提取、HTML 清理等辅助操作                  |
| LLM 接口   | ARK Responses API           | httpx 直调 `/api/v3/responses`，不依赖任何 SDK      |
| MD 渲染    | `mistune`（继承 HTMLRenderer） | 自定义渲染器，直接输出带内联样式的 HTML           |
| 日志       | `logging` 标准库            | 文件 + 控制台双输出，记录全流程                     |
| 环境变量   | `python-dotenv`             | 从 `.env` 文件加载敏感配置                          |
| 输出格式   | 内联样式 HTML               | 微信公众号编辑器不支持外部 CSS，必须使用内联样式     |

### 4.2 LLM 处理模块 (`llm_processor.py`)

#### 职责
- 将爬虫抓取的结构化数据组装为 LLM 输入
- 使用 httpx 直调 ARK Responses API（不依赖任何第三方 SDK）
- 限量发送图片给 LLM 进行图文理解（控制 Token 成本）
- 解析 LLM 输出为结构化的优化后内容

#### API 对接

**放弃 OpenAI SDK**，使用 httpx 直接 POST 请求，避免 SDK 适配问题，后续换模型也无缝迁移：

```python
import httpx

async def call_ark_api(payload: dict) -> dict:
    """直接调用 ARK Responses API"""
    async with httpx.AsyncClient(timeout=120.0) as client:
        response = await client.post(
            f"{ARK_BASE_URL}/responses",
            headers={
                "Authorization": f"Bearer {ARK_API_KEY}",
                "Content-Type": "application/json",
            },
            json=payload,
        )
        response.raise_for_status()
        return response.json()
```

#### ARK API 格式说明

**重要**：ARK 提供两套 API，格式不同，不可混用。

| 对比项            | Responses API                       | Chat Completions API                |
| ----------------- | ----------------------------------- | ----------------------------------- |
| 端点              | `/api/v3/responses`                 | `/api/v3/chat/completions`          |
| 顶层字段          | `input`                             | `messages`                          |
| 文本类型          | `type: "input_text"`                | `type: "text"`                      |
| 图片类型          | `type: "input_image"`               | `type: "image_url"`                 |
| 图片URL格式       | `image_url: "https://..."`（扁平）  | `image_url: { url: "..." }`（嵌套） |
| 响应结构          | `output[]`                          | `choices[].message`                 |
| Token统计         | `input_tokens` / `output_tokens`    | `prompt_tokens` / `completion_tokens`|

**本项目使用 Responses API**，请求示例（`example_payload.json`）：

#### 图片 Token 成本控制

**核心策略**：不把所有图片都传给 LLM，只传前 N 张关键图，其余仅保留位置标记。

```python
# 配置项
MAX_SEND_IMG_NUM = int(os.getenv("MAX_SEND_IMG_NUM", "3"))  # 默认只传前 3 张

# 构造输入时

remaining_images = article_data.images[MAX_SEND_IMG_NUM:]  # 其余只留标记
```

对于未发送给 LLM 的图片，在文本输入中标记 `[图片N]`，确保输出中保留位置，后续由格式化模块插入。

> **关于智能图片选择**（优先传含文字/图表的图片）：首版不实现。原因：需要额外的模型调用判断图片类型，增加复杂度和成本，且前 N 张通常已覆盖文章关键配图。后续版本可考虑。

### 4.3 格式化模块 (`formatter.py`)

#### 职责
- 使用自定义 Markdown 渲染器，**一步到位**生成带内联样式的微信 HTML
- 在 `[图片N]` 位置插入对应图片的 `<img>` 标签
- 在 `[视频N]` 位置插入视频占位符
- 不做后处理字符串替换，避免 bug

#### 设计思路：自定义 MD 渲染器

**继承 `mistune.HTMLRenderer`**（而非 BaseRenderer），保留默认 HTML 渲染行为，仅 override 需要自定义样式的元素：

```python
import mistune

class WeChatRenderer(mistune.HTMLRenderer):
    """微信公众号内联样式渲染器，继承 HTMLRenderer 保留默认行为"""

    NAME = "wechat"

    def paragraph(self, text):
        style = wechat_style.PARAGRAPH
        return f'<p style="{style}">{text}</p>\n'

    def heading(self, text, level, **attrs):
        if level == 1:
            style = wechat_style.H1
        elif level == 2:
            style = wechat_style.H2
        elif level == 3:
            style = wechat_style.H3
        else:
            style = wechat_style.HEADING_DEFAULT
        return f'<h{level} style="{style}">{text}</h{level}>\n'

    def block_quote(self, text):
        style = wechat_style.BLOCKQUOTE
        return f'<blockquote style="{style}">{text}</blockquote>\n'

    def image(self, text, url, title=None):
        style = wechat_style.IMAGE_WRAPPER
        img_style = wechat_style.IMAGE
        return (f'<p style="{style}">'
                f'<img src="{url}" alt="{text}" style="{img_style}" />'
                f'</p>\n')

    def link(self, text, url, title=None):
        style = wechat_style.LINK
        return f'<a href="{url}" style="{style}">{text}</a>'

    def list(self, text, ordered, **attrs):
        tag = "ol" if ordered else "ul"
        style = wechat_style.LIST
        return f'<{tag} style="{style}">\n{text}</{tag}>\n'

    def list_item(self, text, **attrs):
        style = wechat_style.LIST_ITEM
        return f'<li style="{style}">{text}</li>\n'

    def codespan(self, text):
        style = wechat_style.CODE_INLINE
        return f'<code style="{style}">{text}</code>'

    def block_code(self, code, info=None):
        style = wechat_style.CODE_BLOCK
        return f'<pre style="{style}"><code>{mistune.escape(code)}</code></pre>\n'

    def table(self, text):
        style = wechat_style.TABLE
        return f'<table style="{style}">\n{text}</table>\n'

    def emphasis(self, text):
        style = wechat_style.EMPHASIS
        return f'<em style="{style}">{text}</em>'

    def strong(self, text):
        style = wechat_style.STRONG
        return f'<strong style="{style}">{text}</strong>'

    def thematic_break(self):
        style = wechat_style.HR
        return f'<hr style="{style}" />\n'
```

#### `[图片N]` 和 `[视频N]` 的处理

在 MD 渲染之前，预处理 LLM 输出：

```python
def preprocess_markdown(md_text: str, images: list, videos: list) -> str:
    """将 [图片N] 和 [视频N] 标记替换为标准 MD 图片/HTML 占位符"""
    for img in images:
        # [图片1] → ![图片1描述](图片URL)
        md_text = md_text.replace(f"[图片{img.index}]", f"![{img.alt}]({img.url})")

    for video in videos:
        # [视频1] → HTML 视频占位符（原样保留，不经过 MD 渲染）
        placeholder = (
            f'<div style="text-align:center;margin:15px 0;padding:30px;'
            f'background-color:#f0f0f0;border-radius:8px;border:1px dashed #ccc;">'
            f'<p style="font-size:16px;color:#666;">📹 视频位置</p>'
            f'<p style="font-size:12px;color:#999;">请在微信公众号编辑器中手动插入视频</p>'
            f'<p style="font-size:12px;color:#999;">原始链接：{video.url}</p>'
            f'</div>'
        )
        md_text = md_text.replace(f"[视频{video.index}]", placeholder)

    return md_text
```

渲染流程：
```
LLM 输出 Markdown
    ↓
preprocess_markdown()：[图片N] → 标准 MD 图片语法，[视频N] → HTML 占位符
    ↓
mistune 自定义渲染器：直接输出带内联样式的 HTML
    ↓
包裹外层 section + 头部注释
    ↓
最终 HTML（无需后处理）
```

#### 最终 HTML 包裹

```html
<!--
  微信公众号文章 - 由 WeChatArticleAutoGen 自动生成
  原始链接：{url}
  生成时间：{datetime}
  使用方式：...
-->
<section style="max-width:100%;margin:0 auto;padding:0 10px;">
  {渲染后的HTML内容}
</section>
```

#### 图片 URL 有效性校验（可选）

在输出 HTML 前，可对图片 URL 做 HEAD 请求校验：

```python
async def check_image_urls(images: list[Image]) -> dict[str, bool]:
    """检查图片 URL 是否可访问，返回 {url: is_valid}"""
    result = {}
    async with httpx.AsyncClient(timeout=10.0) as client:
        for img in images:
            try:
                resp = await client.head(img.url, follow_redirects=True)
                result[img.url] = resp.status_code == 200
            except Exception:
                result[img.url] = False
    return result
```

> 配置项 `CHECK_IMAGE_URLS=true` 开启，默认关闭（批量处理时增加耗时）。

#### 微信公众号 HTML 规范

微信公众号编辑器的限制：
1. **不支持外部 CSS**：所有样式必须内联
2. **不支持 `<script>`**：无法嵌入 JS
3. **不支持 `<iframe>`**：无法嵌入外部视频播放器
4. **图片自动转存**：粘贴后微信会自动将图片转存到微信 CDN
5. **视频需手动上传**：无法通过 HTML 自动嵌入视频

#### 图片处理详细策略

| 场景               | 处理方式                                                         |
| ------------------ | ---------------------------------------------------------------- |
| 图片原始 URL 可访问 | 保留原始 URL 嵌入 `<img src="...">`，粘贴到微信后自动转存        |
| 图片原始 URL 失效   | 输出时标记 ⚠️ 图片链接失效，需手动替换                            |
| 图片含水印         | LLM 优化时可提示"建议去除水印"，但工具本身不做去水印处理          |
| GIF 动图           | 保留原始 URL，微信编辑器支持 GIF                                  |
| 图片过大（>5MB）   | 输出时标记 ⚠️ 图片过大，建议压缩后替换                            |

#### 视频处理详细策略

| 场景                       | 处理方式                                                              |
| -------------------------- | --------------------------------------------------------------------- |
| `<video>` 标签内的视频     | 提取视频源 URL，输出视频占位符 + 链接，需手动上传                     |
| iframe 嵌入（B站/YouTube） | 提取原始链接，输出占位符 + 链接，需手动上传                           |
| 短视频平台链接             | 提取原始 URL 并记录，供手动处理                                       |

> 视频不做自动下载上传：权限复杂、违规风险高，保持占位符手动处理。

### 4.4 工具函数 (`utils.py`)

#### 文件名安全处理

```python
import re

def sanitize_filename(title: str, max_length: int = 80) -> str:
    """
    清理文件名中的特殊字符，限制长度
    - 替换 \/:*?"<>| 为下划线
    - 去除首尾空白和点号
    - 截断超长标题
    """
    # 替换不安全字符
    name = re.sub(r'[\\/:*?"<>|]', '_', title)
    # 替换中文标点（可选保留，这里替换容易出问题的）
    name = re.sub(r'[？！：、""《》【】]', '_', name)
    # 去除首尾空白和点号
    name = name.strip('. ')
    # 限制长度
    if len(name) > max_length:
        name = name[:max_length].rstrip('_ ')
    # 空标题兜底
    return name or "untitled"
```

#### 文件名冲突处理

```python
def get_unique_filepath(directory: str, filename: str, ext: str) -> str:
    """获取不冲突的文件路径，同名文件自动追加序号"""
    filepath = os.path.join(directory, f"{filename}{ext}")
    if not os.path.exists(filepath):
        return filepath
    i = 2
    while os.path.exists(os.path.join(directory, f"{filename}_{i}{ext}")):
        i += 1
    return os.path.join(directory, f"{filename}_{i}{ext}")
```

import logging
import os
from datetime import datetime

def setup_logger(name: str = "app") -> logging.Logger:
    """初始化日志器"""
    logger = logging.getLogger(name)
    logger.setLevel(logging.DEBUG)

    # 文件输出：logs/app_2026-04-29.log
    log_dir = "logs"
    os.makedirs(log_dir, exist_ok=True)
    log_file = os.path.join(log_dir, f"{name}_{datetime.now().strftime('%Y-%m-%d')}.log")
    file_handler = logging.FileHandler(log_file, encoding="utf-8")
    file_handler.setLevel(logging.DEBUG)

    # 控制台输出
    console_handler = logging.StreamHandler()
    console_handler.setLevel(logging.INFO)

    # 格式
    fmt = logging.Formatter("[%(asctime)s] %(levelname)s %(name)s - %(message)s", datefmt="%Y-%m-%d %H:%M:%S")
    file_handler.setFormatter(fmt)
    console_handler.setFormatter(fmt)

    logger.addHandler(file_handler)
    logger.addHandler(console_handler)
    return logger

#### 异步与并发控制

批量处理时使用 `asyncio` + `Semaphore` 控制并发：

```python
import asyncio

MAX_CONCURRENT = int(os.getenv("MAX_CONCURRENT", "3"))  # 最大并发数

async def process_urls(urls: list[str]):
    """批量处理 URL，控制并发"""
    semaphore = asyncio.Semaphore(MAX_CONCURRENT)

    async def process_with_semaphore(url: str):
        async with semaphore:
            try:
                await process_single_url(url)
            except Exception as e:
                logger.error(f"处理失败 [{url}]: {e}")

    tasks = [process_with_semaphore(url) for url in urls]
    await asyncio.gather(*tasks, return_exceptions=True)
```

#### 执行流程

```
1. 解析命令行参数
2. 加载 URL 列表（单 URL 或文件）
3. 异步批量处理每个 URL（Semaphore 控制并发）：
   a. 调用 crawler.crawl(url) 抓取内容
   b. 调用 llm_processor.process(article_data) 进行 LLM 优化
   c. 调用 formatter.format(optimized_content, article_data) 格式化输出
   d. 保存文件：
      - result/2026-04-29/文章标题.html  （微信公众号用）
      - result/2026-04-29/文章标题.md    （Markdown 备份）
4. 输出汇总结果给用户
```

## 7. 异常处理

| 异常场景             | 处理方式                                           |
| -------------------- | -------------------------------------------------- |
| URL 无法访问         | 返回错误提示，建议检查 URL 或网络，日志记录状态码  |
| 页面无法提取正文     | 多级回退提取器均失败后报错，建议检查 URL 是否为文章页面 |
| LLM API 调用失败    | 重试 3 次（间隔 2s/4s/8s），日志记录每次失败原因   |
| LLM 输出格式异常    | 尝试解析，失败则回退使用原文                       |
| 图片 URL 失效       | 开启校验时在 HTML 中标记失效，不阻断流程           |
| 文件名含特殊字符     | sanitize_filename 清理，限制长度                   |
| 同名文件已存在       | 自动追加序号，如 `文章标题_2.html`                 |
| 批量处理单个URL失败  | 记录错误，继续处理后续URL，不中断批量任务          |

## 9. 已确认决策

| 事项             | 决策                                                                 |
| ---------------- | -------------------------------------------------------------------- |
| LLM 接口        | 固定使用 ARK 原生 `/api/v3/responses` 接口，httpx 直调，不兼容 Chat Completions |
